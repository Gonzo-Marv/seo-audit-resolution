In real-world terms: the home page now appears usable on a phone in 2.3 seconds instead of 10.6 — well inside the threshold where visitors stay instead of bouncing, and inside the band Google rewards in mobile search results. Net positive: orphaned-but-discovered beats orphaned-and-invisible. As internal cross-linking matures, the depth distribution will normalize.

E.02

2 broken internal links

Error ✅ Resolved
About this issue. Same root cause as E.01 — Semrush flags broken internal links as a separate issue category from 4xx pages, but the two findings traced to the same two outdated slug references.

Resolved as a side effect of the E.01 fix.

E.03

12 duplicate meta descriptions

Error ✅ Resolved
About this issue. Pages sharing identical meta descriptions are harder for Google to differentiate in SERPs and frequently result in Google generating its own snippet instead.

Added per-route metadata exports to 6 client-only pages (split into page.tsx server wrapper + *Client.tsx client component pattern). Each route now exports a unique description string at build time. Verified clear in the 17:16 UTC mid-run audit.

W.01

966 disallowed internal resources (robots.txt)

Warning ✅ Resolved
About this issue. The previous public/robots.txt contained Disallow: /_next/, which silently blocked every Next.js JS bundle, CSS file, font, and build-time image asset from being crawled. It also disallowed six real user-facing routes that should have been crawlable.

Rewrote public/robots.txt from scratch. Removed Disallow: /_next/ entirely. Removed spurious Disallow: lines for /smart-security/, /equipment/, /thank-you/, /referral/, /qmerit/, and /order-confirmation/ — all real user-facing pages. Kept /api/, /admin/, and /internal/ blocked.

This was the single highest-leverage change in the entire cycle. Verified clear in the 17:16 UTC mid-run audit.

Single biggest mover

This one fix resolved 1,065 of 1,086 baseline warnings — 89% of every warning in the baseline audit, before any other change shipped. It also auto-resolved a related "14 disallowed external resources" finding (W.02) because the old robots.txt rules were blocking external resource references too.

W.04

12 duplicate title tags → 2

Warning ✅ Resolved (10 of 12)
About this issue. Multiple pages sharing the same <title> tag. Same root cause as duplicate meta descriptions — client-only pages with no per-route metadata export.

Added per-route metadata exports to the 6 client-only pages (split page.tsx + *Client.tsx). Added alternates: { canonical: '/blog' } to consolidate /blog?category=X query-string variants.

The 2 remaining duplicates are the home page found at both / and https://www.smartsecurityconcierge.com/ (non-www / www variant) — Semrush is technically right but resolving requires a server-side redirect or canonical adjustment. Deferred.

W.05

5 pages duplicate content (?category= variants)

Warning ✅ Resolved
About this issue. The blog index renders different category filters via query string (/blog?category=Home+Security+Systems, etc.). Without a canonical, Semrush counted each variant as a duplicate of the unfiltered /blog page.

Same fix as W.04 — the canonical tag on /blog consolidates all 5 query-string variants. Semrush re-crawl confirmed 0.

W.06

20 pages with title tags over 70 characters

Warning ✅ Resolved
About this issue. Semrush flags title tags over 70 characters. Google truncates titles at roughly that length in SERPs but still considers the full text for ranking.

Added an optional metaTitle field to content/posts-metadata.json for 64 of 65 posts. The blog post template now uses metaTitle ?? `${post.title} | Smart Security Concierge` so every effective <title> stays at or under 70 chars while the H1 keeps the full descriptive title.

This is the same trade-off pattern used in the Move Concierge audit — accept long titles for the topical clarity of the H1, but keep the SEO title compact via a separate field.

W.07

59 pages with duplicate H1 = title → 2

Warning ✅ Resolved (57 of 59)
About this issue. Every blog post was rendering the same string in both the <h1> and the <title> tag. Google's guidance is that title and H1 should reinforce the same topic with distinguishable text — not repeat character-for-character.

Changed generateMetadata in app/blog/[slug]/page.tsx to return `${post.title} | Smart Security Concierge` (or the optional shorter metaTitle) for the <title>, while keeping the H1 as {post.title}. Title and H1 now read different strings on every post.

The 2 remaining are non-blog routes flagged for a different underlying reason and tracked separately.

Per-fix detail

What actually moved the score.

P.01

Enable Vercel image optimization

Performance ✅ Resolved
What was wrong. next.config.mjs had images: { unoptimized: true } — a v0 default. Every hero image was served at full source resolution. personalize-system-hero.jpeg was 4406×3524 pixels for a 777×518 display area: 1,667 KiB of waste across 5 images.

Removed the unoptimized: true line. Vercel's image CDN now generates 10 responsive AVIF/WebP variants (256w → 3840w) per image; mobile browsers download the smallest variant that satisfies the viewport. The Hero components already had correct sizes props — they just weren't being honored.

Impact: mobile LCP 10.6s → 7.3s (−3.3s, ~31%). 1,667 KiB savings on first load.

P.02

Defer Google Tag Manager (lazyOnload)

Performance ✅ Resolved
What was wrong. GTM was loaded as an inline <script> in <head>, render-blocking with ~294 KiB of JS (127 KiB unused). On mobile under 4× CPU throttle, this added 150–200ms to TBT before any user interaction.

Wrapped GTM in <Script strategy="afterInteractive"> first, then escalated to <Script strategy="lazyOnload"> after measuring a mid-run regression. Lazy means GTM now loads on browser idle (~3–5 seconds after page paint) rather than during render.

Impact: mobile TBT 590ms → 110ms (−480ms, −81%). Mobile Performance score 59 → 98 (+39 in this single change after the previous fix had exposed GTM as the new bottleneck).

P.03

Defer Convert Experiences (A/B testing)

Performance ✅ Resolved
What was wrong. Loaded as a regular <script src> in <head>, blocking 1,200ms on the critical path with 53 KiB transfer.

Wrapped in <Script strategy="lazyOnload">. Removed Convert from the critical path entirely.

P.04

Remove Google Maps from global layout

Performance ✅ Resolved
What was wrong. <GoogleMapsScript /> was mounted in app/ClientLayout.tsx, meaning every page on the site fetched /api/google-maps-key and loaded the Maps SDK — even though Maps is only used on /order-confirmation and partner pages. The /api/google-maps-key call was on the critical path at 855ms latency.

Removed <GoogleMapsScript /> from ClientLayout.tsx. The two pages that actually use Maps already mount the script themselves.

Impact: removed 855ms of critical-path latency on the homepage and 14 other routes that didn't use Maps.

P.05

Modern-browser JS targets (drop ES2020+ polyfills)

Performance ✅ Resolved
What was wrong. Next.js's SWC compiler was shipping ES2020+ polyfills (Array.prototype.at, .flat, .flatMap, Object.fromEntries, Object.hasOwn, String.prototype.trimStart/trimEnd) to all browsers — about 12 KiB of unused JS per page.

Added .browserslistrc with last 2 versions, not dead, not IE 11, not op_mini all to drop polyfills targeted at obsolete browsers.

Impact: −12 KiB JS bundle, faster JS parse on mobile CPU.

P.06

Add <main> landmark for accessibility

Accessibility ✅ Resolved
What was wrong. Lighthouse accessibility audit flagged "Document does not have a main landmark." Score was 98.

Wrapped {children} in <main> in app/ClientLayout.tsx. Renamed a legacy <main> in app/aspire_mortgage_advisors/page.tsx to <div> to avoid nesting.

Impact: Mobile Accessibility 98 → 100.

Things tried that didn't make the cut

Source-image resizing. Considered using sharp to physically resize the 4400×3500 source JPEGs in public/images/. Skipped because Vercel's image CDN already produces correctly-sized output variants — the only savings would be a one-time speedup on Vercel cold-cache warm-up.

AVIF source conversion. Same reasoning — Vercel auto-converts to AVIF/WebP at output regardless of source format.

Critical CSS inlining. Would save ~380ms on mobile render-blocking but requires either a third-party tool integration or manual extraction. De-scoped — multi-day work for ~5 points of speed-score gain on top of an already-98 score.

Removing Microsoft Clarity. It's loaded indirectly via GTM. Removing it requires editing GTM tags rather than codebase changes. Out of scope for this engagement.

Cluster C · 15 posts

Prepper / second-amendment community.

Self-identified preppers, second-amendment community, current/former military and law enforcement, and adjacent serious-defense readers. Skeptical of "peace of mind" home-security marketing; respond to operational language they recognize from training, doctrine, and milblog culture. Lower SEO competition than mainstream "best home security" terms (the big publishers don't write to this register), high commercial intent (this audience already spends real money on defense gear), and topical authority that cross-pollinates back to mainstream rankings via natural backlinks from forum and blog citations.

Cluster D · 10 posts

Snowbird / vacation-home owner community.

Retired or semi-retired professionals, typically 60–75, who own a primary residence up north and a second home in the Sun Belt. Affluent but value-conscious. Practical, slightly weary of vendor sales pitches, respect their own experience. Underserved by mainstream home-security publishers (most write to "young family buying their first home," not "65-year-old with two homes and an insurance crisis"). Florida specifically is in a real estate / insurance crisis cycle — post-hurricane-season carrier non-renewals, HB 621 squatter law, vacancy clauses tightening — a moment of high search intent for our exact services.

3 hubs Lifecycle · Risk · System

Three intersecting sub-clusters around the snowbird user journey.

Audience-research-first methodology. Before writing any articles, half a day went to producing an audience brief: a 32-term vocabulary glossary, a 12-pain-point ranking, verified authority citations, and an SEO opportunity scan. The brief established voice rules — peace of mind is acceptable register here (unlike the prepper community), SHTF / tactical / militia language is forbidden, and the reader should be addressed as someone who has owned the home for years and doesn't need basic explanation. Every Lifecycle post links to all Risk + System hubs (because users planning a departure want to know the risks and the systems). Risk and System posts cross-link within their cluster and back to the relevant Lifecycle hub.

Production methodology (how 10 posts shipped in one afternoon)

One audience-research agent built the brief. Ten writing sub-agents then wrote one post each in parallel, fed identical voice rules and a cluster-aware internal-link map (each agent knew which other 4 cluster siblings to link to). All 10 posts shipped in a single afternoon. The same fan-out pattern (with batch-specific voice rules) was used for the prepper cluster.
